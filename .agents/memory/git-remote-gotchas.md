## gitPush tool refuses to run if ANY git credential helper is configured
It errors `DANGEROUS_CONFIG` ("credential helper is configured... could access the bearer token") even if the helper is scoped to a different host (e.g. `credential.https://gitlab.com.helper` blocked a GitHub push). Remove any custom credential helper from git config before relying on the `gitPush`/`gitPull`/`createPullRequest` tools; only add one back temporarily for a manual (non-tool) push, then remove it again.

## GitLab push/pull has no first-class tool support — do it with a manual PAT + scoped credential helper
`gitPush`/`gitPull` explicitly reject `gitlab` as a provider (`UNSUPPORTED_PROVIDER`), and Replit's own native "Git pane" GitLab push flow (the in-app "Pass GitLab Credentials" prompt) also currently fails with an UNAUTHENTICATED error — this is a real, currently-unresolved platform gap, not something fixable by reconfiguring the repo. The account-level GitLab connector (via `searchIntegrations`) is also read-only (`read_api, read_repository` scopes only) — it cannot push either.

**Working alternative:** request a GitLab Personal Access Token from the user (`write_repository` scope) via `requestSecrets`, then build a tiny credential-helper script that echoes `username=oauth2` / `password=$GITLAB_TOKEN` from the env var (never print the token yourself), and scope it with `git config credential.https://gitlab.com.helper '!/path/to/script'`. Push with plain `GIT_TERMINAL_PROMPT=0 git push <remote> <branch>` (needs a long timeout — a couple hundred MB of LFS objects took >25s and needs ~90s+). Remove the helper again afterward so the `gitPush` tool isn't blocked (see entry above).

## `origin` remote must be HTTPS for the gitPush tool, not SSH
An SSH-format origin (`git@github.com:owner/repo.git`) is rejected outright ("origin remote must be an HTTPS URL"). Always add/set origin as `https://github.com/owner/repo.git`.

## `UNKNOWN_NOT_GIT` from gitPush on GitHub — unresolved, intermittent
Hit repeatedly on this repo even with a correctly-configured HTTPS `origin`, upstream tracking set, and no credential helper present — tried multiple genuinely different fixes (remote re-add, upstream re-link, retries) and it kept recurring. Root cause undetermined; may be specific to repos with many auto-generated `subrepl-*` checkpoint remotes alongside a manually added `origin`. If hit again, don't sink more time into guessing — tell the user directly and suggest they re-authenticate GitHub from Replit's account-level Git Providers settings, or try the native Git pane's push button, since that's outside what repo-level git config can fix.

## Platform auto-sync repeatedly runs `pull --quiet --no-edit --rebase origin main` and gets stuck — recurs even after unsetting upstream, triggered by task "Apply changes" too
When `origin` points to a GitHub repo whose history is nearly empty (just a placeholder commit) while local `main` is 70+ commits ahead, some Replit-internal process runs `git pull --rebase origin main`, which tries to replay the entire real project history onto that placeholder. It has repeatedly gotten stuck mid-replay (corrupted rebase bookkeeping, `done`/`git-rebase-todo` disagreeing, or just very slow — 70+ steps). While stuck, the live working tree sits at an arbitrary historical checkpoint, which makes the platform's artifact/workflow auto-detection think artifacts were deleted (it briefly deregisters them, then re-registers them once the rebase is resolved and the real files are back — no actual data loss, just alarming transient UI churn).

**Confirmed triggers:** (1) whatever periodic background sync ran previously, and (2) using the mobile Git pane's project-task "Apply changes"/merge flow — even for a task whose own diff is unrelated (e.g. a "clean up stale git remotes" task with a `.git/config`-only change) — appears to invoke the same `pull --rebase origin main` as a reconciliation step before merging the task branch.

**`git branch --unset-upstream main` is NOT a durable fix** — the platform re-establishes `main` tracking `origin/main` on its own (confirmed: it came back after a later task-merge attempt with no manual action). Don't rely on local git config to suppress this.

**Recovery recipe (safe, repeatable):** `git rebase --abort`. If it fails with "untracked working tree files would be overwritten" (commonly `.replit`), `mv` the untracked file out of the way first, then abort, then decide whether to keep or discard the moved-aside file (diff it against the restored tracked version — it's often just a stale copy from deep in history and safe to discard). Recheck `git status`/`ls artifacts/` after — everything is normally intact.

**Underlying fix is out of repo-config reach:** the real fix is getting an actual GitHub push to succeed (see `UNKNOWN_NOT_GIT` above) so local and origin converge and there's nothing left to rebase. Until then, expect this to recur on any task-merge/apply-changes action, and treat it as a known, recoverable, cosmetic-looking incident rather than data loss.

## `UNKNOWN_NOT_GIT` is specific to the git-push code path — GitHub API auth (createPullRequest) works fine
Diagnosed further on this repo: `gitPush` fails with `UNKNOWN_NOT_GIT` on every branch tried (not just `main` — a disposable throwaway branch pushed identically failed the same way), ruling out branch-name or content causes. But `createPullRequest` on the same repo successfully reaches GitHub's GraphQL API (it got as far as a real API response, `"No commits between main and main"`, rather than any auth/credential error) — proving the account-level GitHub connection itself is valid and authenticated. Also confirmed independently: `https://github.com/<owner>/<repo>.git/info/refs?service=git-receive-pack` returns a normal `401 No anonymous write access` (correct git smart-HTTP behavior, not a broken/non-git response), and `git fsck`/`git count-objects` show no repo corruption. This narrows the bug to specifically how the `gitPush` tool wires credentials into the underlying `git push` subprocess (separate from the `gh` API auth `createPullRequest` uses) — not something fixable from repo config, remote URL format, or branch naming. If this recurs, don't re-litigate those already-ruled-out causes; just report it as a tool-level bug and point the user at re-authenticating GitHub in Replit's account Git Providers settings, or Replit support.
